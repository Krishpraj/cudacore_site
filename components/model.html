What is the "forward pass"?

One run of the model from input to output. You hand it a list of token IDs, and it hands back a vector of scores. one score for every possible next token in the vocabulary. The token with the highest score (greedy sampling) is what the model "wants" to say next.

¶
analogy

Picture an assembly line, 22 stations long. A vector of numbers travels down the line. Each station tweaks it a little. normalizes it, mixes information from earlier tokens, runs it through a small neural network, adds the result back to the original. By the time it reaches the end, the model has computed what should come next.

The shape of the data, end to end

  1. Tokens in. [10, 1456, 92, 3]: four token IDs.
  2. Embedding lookup. Each ID picks out one row of a giant [vocab_size, hidden_dim] table. Now you have a [4, hidden_dim] matrix. one vector per token.
  3. The 22 layers. Each layer applies attention then MLP, with residual adds. The shape [4, hidden_dim] never changes. only the numbers in it do.
  4. Final norm. One last RMSNorm on the last position only.
  5. LM head. A final matmul that projects from hidden_dim up to vocab_size. You now have a score for every possible next token. Done.

What happens inside one layer

The two halves of every transformer block: an attention sublayer that mixes information across positions, then an MLP sublayer that thinks position-by-position.

Attention. "what should I pay attention to in the past?"

x_norm = RMSNorm(x)

q = W_q · x_norm     // "what am I looking for?"
k = W_k · x_norm     // "what am I about?"
v = W_v · x_norm     // "what would I contribute?"

q, k = apply_RoPE(q, k, position)

scores = q · K^T / sqrt(head_dim)   // similarity of this q to every past k
scores = scores + causal_mask        // can't look at the future
weights = softmax(scores)
out = weights · V                    // weighted sum of past v's

attn = W_o · out
x = x + attn                         // residual add

Three things to take from this:

  • Each token computes a query, a key, and a value from itself. The query says "what am I curious about." The key says "what I represent." The value says "what I'd contribute if you decided I'm relevant."
  • The model computes the similarity of this token's query against every past token's key, softmaxes those into weights, then mixes their values together.
  • The causal mask ensures token t only sees tokens 0..t. This is what makes the model autoregressive.

MLP. "now think about it alone"

x_norm = RMSNorm(x)

gate = W_gate · x_norm
up   = W_up   · x_norm
mlp  = W_down · ( silu(gate) * up )    // SwiGLU
x = x + mlp                            // residual add

No mixing across positions here. Each token is reprocessed independently through a 2-layer feed-forward network. The hidden dim doubles or triples inside before projecting back down. that wider middle is where most of the model's parameters live.

Grouped-query attention (GQA), in one paragraph

Llama-2 70B and Llama-3 reduce KV cache size by sharing key/value heads across multiple query heads. If you have 32 Q-heads and 8 KV-heads, every group of 4 Q-heads reads from the same K and V. The model gets the expressiveness of 32 query views but only pays for 8 keys/values. For TinyLlama (the easy starting model) num_heads == num_kv_heads and GQA collapses to standard multi-head attention.

Loading real weights

HuggingFace ships models as model.safetensors: a JSON header with tensor names, shapes, and dtypes, followed by raw bytes. Tiny format, takes a weekend to write a parser. The cudacore loader handles fp32 single-shard files and binds about a dozen named tensors per layer to in-memory pointers.

How to know it's correct

Don't trust the model until intermediate activations match a Python reference within fp32 tolerance:

  1. In Python: AutoModelForCausalLM, run one forward, dump every layer's hidden state to .npy.
  2. In C++: add a debug hook that writes x after each layer's residual add.
  3. np.allclose(yours, theirs, atol=1e-4). The first layer that disagrees is where the bug is.
  4. End-to-end DoD: 50 greedy tokens identical to generate(do_sample=False). Identical, not "similar."

what i learned

  • Matrix layout decides your sanity. Row-major vs column-major; Linear weights as (out, in) vs (in, out). Pick a convention, write it down, never deviate. Most "garbage output" bugs are a silently transposed matmul.
  • The 7th token rule. If your output diverges from HuggingFace at token 7, the bug is real. Fix it now. Debugging a wrong forward pass at Stage 5 with batching on top is impossible.
  • RoPE is the bug. If outputs look almost right but slowly drift, it's RoPE. Either the pairing convention is wrong, or you forgot to scale theta correctly per dim.
  • Tied embeddings save 100MB. Many small models reuse the embedding matrix as the LM head (tie_word_embeddings: true). Half the checkpoints don't include lm_head.weight at all. you're supposed to alias it to embed_tokens.weight.
  • No KV cache yet is fine for now. Stage 2's job is correctness. Recomputing all past tokens on every step is wasteful, but it's also the simplest possible code path, and you want simple here.
  • The "block" is really tiny. Two matmul stages, two norms, two residual adds. Modern LLMs are almost embarrassingly simple architectures stacked deep with a lot of data behind them.
