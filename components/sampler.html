Sampler

The model gives you a vector of scores — one per possible next token, often 32,000 to 128,000 of them. You have to pick exactly one. How you pick decides whether the model sounds robotic, creative, or unhinged.

What "sampling" actually means here

After the forward pass, the model gives you logits — raw, unnormalized scores. Bigger logit = more likely the model wants that token. To turn this into an actual choice, you have a few strategies, from totally deterministic to very creative.

¶
analogy

The model is a sommelier: it ranks every possible word from "definitely yes" to "definitely no." The sampler is you, deciding how to actually pick. Always grab the top pick? You'll be predictable. Pick randomly from the top 40? You'll be more interesting but never weird. Weight a coin by the rankings and flip? Closer to how a person actually speaks.

The four strategies

Greedy — always pick the top

next = argmax(logits)

Deterministic. Same prompt always produces the same output. Good for: testing correctness, code, math, structured output. Bad for: creative writing — gets repetitive and bland because there's no diversity at all.

Temperature — flatten or sharpen the distribution

probs = softmax(logits / T)
next = sample_from(probs)

Divide logits by T before softmax. T → 0 recovers greedy. T = 1 uses the model's natural distribution. T > 1 flattens it (more diversity, more chaos). Most chat assistants live around T = 0.7.

Top-k — keep only the k highest-logit tokens

keep the top k logits, set the rest to -inf
probs = softmax(kept / T)
next = sample_from(probs)

Common k = 40. Bounds the worst-case bizarre choice — the model can't pick a wildcard token from the long tail. Still lets the top candidates compete on their relative merits.

Top-p (nucleus) — keep the smallest set summing to probability p

sort logits descending → softmax
keep tokens until cumulative prob ≥ p (e.g., 0.9)
renormalize, sample

Common p = 0.9–0.95. Adapts to the distribution's shape. On a confident prediction ("the cat sat on the ___" → "mat"), you might keep 3 tokens. On an open one ("once upon a ___"), you might keep 100. Generally produces the most natural-feeling output, which is why most APIs default to it.

The combined pipeline

In practice, all three filters compose:

logits = model.forward(...)
logits = apply_repetition_penalty(logits, recent_tokens)
logits = top_k_filter(logits, k=40)        // hard cap
logits = top_p_filter(logits, p=0.9)       // soft adapt
probs  = softmax(logits / T)               // temperature
next   = categorical_sample(probs, rng)

Sampling from a probability vector

The classic way:

r = uniform(0, 1)
cum = 0
for i in 0..vocab_size:
    cum += probs[i]
    if r < cum: return i

Linear scan, O(vocab_size). Fine — 128k floats is microseconds compared to the millisecond forward pass. C++ has std::discrete_distribution that does this for you.

Repetition penalty

Optional but commonly expected. Divide the logits of recently-generated tokens by a small factor (usually 1.05–1.20) before sampling. Discourages the model from getting stuck in loops ("the the the the..."). Aggressive penalties hurt quality; mild ones are usually safe.

Where does this run?

On a single [vocab_size] vector. Maybe 30,000–128,000 floats. Microseconds of work compared to the millisecond forward pass. Cheapest part of the pipeline. Do it on the CPU after copying logits back from the GPU. Once you've optimized everything else, there's a minor win in keeping the next-token-id on the GPU to skip a tiny device→host→device hop, but don't start there.

// what I learned

  • Start with greedy. Always. When you're debugging the forward pass, randomness in sampling makes it harder to spot regressions. Greedy = same prompt → same output → easy bisection.
  • Top-p > top-k for most chat. Top-k's "always 40 candidates" is too restrictive when the model is confident, and not restrictive enough when it isn't. Top-p adapts.
  • Numerical stability matters here too. Same softmax-max-subtraction rule. Raw logits can be 30+, exp overflows, NaN propagates, your sampler "picks" the wrong thing silently.
  • Don't softmax twice. Operate on logits everywhere, softmax once at the end. Two softmaxes is a real bug I had once — the output technically sums to 1 but the distribution is wrong.
  • The sampler returns EOS, the caller stops. Don't filter EOS out of the distribution. The model has to be able to emit "I'm done"; you just have to listen for it.
  • Seed the RNG once. Reseeding per call is a common mistake — you get the same "random" choice every step. One std::mt19937 at startup, advance it forever.
