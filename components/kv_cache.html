KV cache

Store the key and value vectors from past tokens so you don't have to recompute them every single step. The single most impactful optimization in autoregressive LLM inference.

The problem it solves

Without a KV cache, generating token N means running the entire forward pass over all N-1 previous tokens, then computing attention for the new one. To generate 100 tokens you've effectively run the model thousands of times. Total work to generate N tokens: O(N²).

But here's the thing: when you generated token 50, you already computed the key and value vectors for tokens 1 through 49. They depend only on the input, and the input hasn't changed. Throwing them away and recomputing them on step 51 is pure waste.

¶
analogy

You're a detective interviewing witnesses to build a case. Without a notebook, every new witness means you'd have to re-interview every previous one too — by witness 50 you'd never sleep. The KV cache is the notebook: write each interview down once, flip back when you need it.

What gets cached, exactly

In attention, each token contributes a query, a key, and a value vector to each layer. Only K and V need to be remembered — Q is consumed immediately to compute that token's attention output and never looked at again.

So per token, per layer, you save:

K_t = W_k · x_t    // shape: [num_kv_heads, head_dim]
V_t = W_v · x_t    // shape: [num_kv_heads, head_dim]

The cache for one layer is just a growing list of these. Total shape:

K_cache: [max_seq_len, num_kv_heads, head_dim]
V_cache: [max_seq_len, num_kv_heads, head_dim]

And you have one of these per layer.

Two phases: prefill vs decode

Prefill — process the whole prompt

The user gives you a prompt of length P. You run the full forward pass once on all P tokens at once. K and V are computed for every position. You shove all of them into the cache in one shot. This phase is compute-bound — one big matrix multiply per layer.

Decode — generate one token at a time

Now the loop kicks in. For each new token:

  1. Forward pass operates on just one new token, not the whole sequence.
  2. Compute Q, K, V for that one token.
  3. Append K and V to the cache.
  4. Attention reads the entire cache for keys/values, but Q is still just one row.
  5. Output one new token, repeat.

This phase is memory-bound. The matmul has Q-rows = 1 — there's barely any compute to do. Most of the time goes to streaming weights and the cache through GPU memory.

How big does it get?

Per token, per layer, the cache stores 2 × num_kv_heads × head_dim × bytes_per_element bytes (×2 for K and V).

TinyLlama 1.1B, fp16:
  num_kv_heads=4, head_dim=64, num_layers=22
  per token: 2 × 4 × 64 × 2 × 22  = 22,528 bytes
  at seq_len=2048:                ≈ 46 MB     (trivial)

Llama-2 70B, fp16:
  num_kv_heads=8, head_dim=128, num_layers=80
  per token: 2 × 8 × 128 × 2 × 80 = 327,680 bytes
  at seq_len=4096:                ≈ 1.3 GB    (suddenly significant)

For long-context serving (32k, 128k tokens), the cache can dwarf the model weights themselves. This is why "PagedAttention" exists in Stage 7 — to avoid pre-allocating the worst case.

Memory layout — the one decision that matters

Two reasonable choices for how to lay out K in memory:

[seq, head, dim] — natural for appending. A new token's K is contiguous; you just write the next slot.

[head, seq, dim] — natural for the attention matmul. Each head's keys are contiguous, which is what the per-head q · K^T needs.

Production runtimes use the second. The append becomes scattered writes (small cost per token) but the read — which happens way more often — gets the contiguous layout (big payoff). For a Stage 3 reference, either is fine. Pick one once you understand the tradeoff.

The API, in pseudo-code

struct KVCache {
    Tensor K;  // [num_layers, max_seq_len, num_kv_heads, head_dim]
    Tensor V;
    int current_len;
};

// Decode-time append:
cache.K[layer, current_len] = k_new;
cache.V[layer, current_len] = v_new;
// (after writing all layers for this token)
cache.current_len += 1;

// Attention read:
K_slice = cache.K[layer, 0 .. current_len];
V_slice = cache.V[layer, 0 .. current_len];
// scores = q @ K_slice^T / sqrt(d_k)
// out    = softmax(scores) @ V_slice

// what I learned

  • It's the biggest single optimization in inference. The first time you turn it on you go from 30s for 50 tokens to 3s. Every other stage's gains are smaller in percentage terms.
  • Off-by-one on current_len is the recurring bug. Increment after the write, not before. Attention reads [0..current_len) — read past that and you're attending to uninitialized garbage which still produces "valid looking" tokens.
  • Pre-allocating max_seq_len is fine for a single-user toy. It's wasteful for serving — a request asking for 50 tokens shouldn't reserve space for 2048. That's the motivation for PagedAttention later.
  • Different sequences need different cache slots. The moment you batch (Stage 7), each sequence has its own cache. You can't share a contiguous block across sequences with different lengths. This is the entire motivation for vLLM's paged design.
  • The "prefill = compute-bound, decode = memory-bound" distinction is the lens for all later optimization. Different bottlenecks need different fixes. Throwing more FLOPs at decode does nothing if you're already saturating DRAM.
