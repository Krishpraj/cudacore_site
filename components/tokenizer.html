Tokenizer

The translator at the door. Turns the string "Hello, world!" into a list of integers the model can actually compute on — and turns the model's output integers back into letters you can read.

What is it, really?

A neural network is just a giant pile of math on numbers. It can't read letters. So before the model touches your prompt, something has to convert "Hello" into something like [15043, 11526]. That something is the tokenizer. After the model finishes, the same tool runs in reverse — integer output back into text.

¶
analogy

Imagine a library where every word, syllable, and punctuation mark has a numbered card. The tokenizer is the receptionist who looks up your sentence and hands you a stack of cards in the right order. The model only ever shuffles the cards — it never sees the words on them.

Why a "token" isn't just a word

You'd think one word = one token. It isn't. Modern tokenizers use Byte-Pair Encoding (BPE) or SentencePiece, which split text into subword pieces. "running" might be one token (common). "Krishpraj" might be three (rare). Punctuation, leading spaces, and emoji each get their own.

Why subwords? Two reasons:

  • Coverage. A pure-word vocabulary can't handle new names, code, or typos. A subword vocabulary can spell anything by combining pieces.
  • Compression. Common things become one token. Rare things become a few. The model sees compressed input, which is shorter and cheaper.

How encoding actually works

The tokenizer ships with a fixed vocabulary file: a list of around 32,000–128,000 string fragments, each with an integer ID. To encode "hello":

  1. Look at your text and find the longest prefix that exists in the vocabulary.
  2. Emit its ID. Move past it. Repeat.
  3. If nothing matches, fall back to an <unk> (unknown) token and skip one character.

Decoding is the reverse: take a list of IDs, look up each one's string fragment, glue them together. The catch — covered below — is that one Unicode character can span multiple tokens, so naive concatenation breaks emoji.

The sneaky bits

Spaces aren't really spaces

SentencePiece (Llama, T5) encodes a space as the character ▁ (U+2581). So " world" becomes "▁world" internally. This is what lets the tokenizer treat "world" at the start of a sentence differently from " world" in the middle, with a single vocabulary entry.

BOS and EOS

Most models want a "beginning of sequence" token (<s>) prepended and an "end of sequence" token (</s>) appended. Get this wrong and the model will produce coherent but slightly off-distribution text.

Streaming decode is harder than it looks

When you generate one token at a time, you can't decode it in isolation — a Japanese character might span two BPE tokens, and decoding one alone gives you half a character (renders as �). You have to buffer and emit only complete UTF-8 sequences. This bites everyone the first time.

The code, in 30 seconds

// Load a vocabulary file
Tokenizer tok("vocab.txt");

// Text in, integers out
std::vector<int> ids = tok.encode("Hello, world!");
// e.g. [1, 15043, 29892, 3186, 29991, 2]

// Integers back to text
std::string out = tok.decode(ids);
// "Hello, world!"
implementation notes

The cudacore tokenizer is a small C++ class that loads a tab-separated vocab file, runs a greedy longest-match search to encode, and reverses the lookup to decode. Special tokens (BOS, EOS, UNK) are auto-detected on load. The hard part wasn't the algorithm — it was getting whitespace, the ▁ marker, and the optional "dummy space prefix" to round-trip exactly the way HuggingFace expects.

// what I learned

  • Test the round-trip first, before anything else. If decode(encode(s)) != s on weird inputs (leading spaces, unicode, code), every later stage gets harder to debug because you'll never be sure if the bug is yours or the tokenizer's.
  • "Tokenizer matches HuggingFace" means byte-for-byte identical. Off by one ID anywhere and a 7B model can output gibberish — the embedding row it looks up is wrong.
  • The ▁ trick is elegant once it clicks. One string fragment, one entry in the vocab, but it carries the "is this word-initial?" bit along with it. No separate punctuation logic needed.
  • Greedy longest-match is the right default for SentencePiece-style vocabs. Real BPE technically applies ranked merges, which behaves slightly differently on tie cases. For learning, longest-match is correct enough.
  • Don't confuse "byte-level BPE" (GPT-2) with "SentencePiece BPE" (Llama). Same name, very different rules around bytes vs. characters. The model's config tells you which one it expects.
