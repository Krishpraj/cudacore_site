Why a GPU at all?

A modern CPU does about 100 GFLOPS of fp32. A consumer GPU (RTX 4070) does ~30 TFLOPS — about 300× more. The catch is the GPU only delivers that throughput if you give it many independent operations to do at once. Matrix multiplies are perfect for this: every element of the output is independent of every other.

¶
analogy

A CPU core is one extremely skilled craftsman — versatile, fast at any one thing, but only one at a time. A GPU is a hangar full of less-versatile workers (10,000 of them) who all do the exact same task at the exact same moment on different pieces of the input. Hand them a matmul: bliss. Hand them an if-statement that branches differently per worker: misery.

The GPU mental model

A CUDA program launches a grid of blocks, each containing many threads. Threads in the same block can share fast on-chip memory; threads in different blocks can't talk to each other except through slow global memory.

Threads execute in groups of 32 called warps, in lockstep. If 31 threads in a warp take the if branch and 1 takes the else, the hardware executes both branches and masks the unused results. This is why branchy code kills GPU performance.

Memory hierarchy

  • Registers — instant. Per-thread.
  • Shared memory — ~kB per block, ~100× faster than global. Explicitly managed: you decide what goes here.
  • L1 / L2 cache — automatic, like CPUs.
  • Global (DRAM) — GBs available but ~600 cycles to access. Most kernels are bottlenecked here.

The order I built it in

1. Plumbing — get tensors onto the device

A DeviceTensor that wraps cudaMalloc / cudaFree. A cudaCheck macro that aborts loudly on any error — silent CUDA failures are misery to debug because the error doesn't surface at the failing call, it surfaces at the next sync. Always sync after each kernel during debugging.

2. cuBLAS for matmuls

I did not write a GEMM from scratch first. cuBLAS is the product of NVIDIA engineers tuning matmul for two decades. Nothing I write will beat cublasSgemm in fp32 or cublasGemmEx in fp16 for the first six months. It's the baseline; my custom kernels exist only where there's no library equivalent.

Watch out: cuBLAS is column-major by default. With row-major tensors, you typically express A @ B as B^T @ A^T with transposed flags. The docs have a whole section on this. Read it twice.

3. Hand-written kernels for the small ops

The fun part. These are the operations cuBLAS doesn't cover:

  • Elementwise — residual add, SwiGLU activation, RoPE rotation
  • Reductions — RMSNorm (mean of squares), softmax (max + sum)
  • Gather — embedding lookup

Each one teaches a CUDA pattern. The reduction pattern is the most reused — warp shuffles for the per-warp reduction, then shared memory to combine across warps.

4. Fused kernels

Memory traffic dominates. Reading x from DRAM costs the same as a hundred FLOPs of arithmetic. So you fuse: do as much work as possible per byte loaded.

  • RMSNorm + the next linear's input prep — read x once instead of twice.
  • Q/K/V projections in one launch — share the read of x across three writes.
  • Attention as a single kernel — never materialize the full score matrix in global memory.

5. FlashAttention

The last one above, done right. FlashAttention v1 keeps the per-block score matrix in shared memory only, tiles the computation, and updates running softmax statistics as it goes. The result: same math as standard attention, but 2–4× faster and using far less memory. Read the paper; many runtimes just link the official implementation rather than rewriting it.

The patterns you'll use over and over

Warp-level reduction (sum across 32 threads)

float v = my_value;
for (int offset = 16; offset > 0; offset /= 2)
    v += __shfl_down_sync(0xffffffff, v, offset);
// thread 0 of each warp now has the warp sum

Coalesced memory access

Adjacent threads in a warp should read adjacent bytes from global memory. Hit this and 32 reads collapse into one transaction. Miss it (strided or scattered) and you waste ~90% of the available bandwidth. Most "why is my kernel slow" questions reduce to "your access pattern isn't coalesced."

The roofline lens

For each kernel, ask: how many FLOPs am I doing per byte read? Plot it. Decode-time matmul (matrix × vector) has very low arithmetic intensity — it's memory-bound, and the fix is fewer bytes (quantization). Prefill-time matmul (matrix × matrix on a long prompt) is compute-bound, and the fix is more FLOPs/cycle (Tensor Cores). Knowing which side of the roofline you're on is everything.

Profiling — non-negotiable

After every change to a kernel, run Nsight Compute on it. The Speed of Light view shows what % of peak compute and peak memory you're achieving. The warp-stall breakdown tells you exactly what's slowing things down:

  • LG Throttle, Long Scoreboard → memory-bound. Cache, tile, fuse.
  • Math Pipe Throttle → compute-bound. fp16 or Tensor Cores will help.
  • MIO Throttle → load/store unit saturation, look at coalescing.

// what I learned

  • cuBLAS first, always. Trying to beat it in week one is a trap. Use it, measure against it, write your own only where it has gaps (small reductions, elementwise, fused ops).
  • Row-major vs column-major bug is unavoidable, plan for it. Add an assert that compares your CUDA output to the CPU reference on a tiny input for every new kernel. Catches transposes in seconds instead of days.
  • Async errors are the worst. A kernel launch error doesn't show up at the launch site — it surfaces at the next cudaDeviceSynchronize or memory copy. While developing: sync after every launch.
  • Occupancy is a metric, not a goal. A kernel with 30% occupancy and great cache behavior beats a 90%-occupancy one thrashing DRAM. Don't optimize occupancy in isolation.
  • Most LLM inference is memory-bandwidth-bound, not compute-bound. This was the biggest mental model shift. The "tflops" number on the GPU spec sheet doesn't matter for decode. The GB/s number does.
  • FlashAttention is a clever algorithm, not a clever kernel. The win is keeping the softmax state online as you tile — that lets you avoid writing the full attention matrix to global memory. The CUDA part is the mechanical implementation.
  • Windows + native CUDA is painful. WSL2 + Ubuntu is a one-evening setup and saves a week.
