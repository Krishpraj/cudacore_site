What is a "kernel"?

In this context, a kernel is just a small math function: take some numbers in, produce some numbers out. Matrix multiply is a kernel. Softmax is a kernel. RMSNorm is a kernel. The word becomes more loaded when we move to CUDA. there a "kernel" is a specific kind of function that runs in parallel on the GPU. but for now, think "small math routine."

¶
analogy

If the model is a recipe, kernels are the basic cooking moves: chop, sauté, whisk, fold. A transformer is just one specific recipe that uses these same five or six moves over and over in different orders. Learn the moves once, you can read every recipe.

The six pieces

Matrix multiply (matmul)

The workhorse. C = A × B. About 95% of the FLOPs in a forward pass come from this one operation, applied dozens of times per layer with different weight matrices. The naive version is three nested loops:

for m in [0..M):
  for n in [0..N):
    sum = 0
    for k in [0..K):
      sum += A[m,k] * B[k,n]
    C[m,n] = sum

That's it. Slow but correct. On CPU you'd link OpenBLAS or MKL for the real version; on GPU you'd call cuBLAS. The hand-rolled version exists so you can be certain the optimized one gives the same answer.

Softmax (numerically stable)

Turns a vector of arbitrary real numbers into a probability distribution. all positive, summing to 1. The trick: subtract the max first.

m   = max(x)
e   = exp(x - m)     // largest value of e is now exp(0) = 1
s   = sum(e)
out = e / s

Why subtract the max? Because attention logits can be 30 or more, and exp(30) is around 10¹³. Sum a few of those and you overflow fp32. Subtracting the max first guarantees the largest term is 1 and everything else is between 0 and 1. This bug is so common and so silent it ranks in every top-five list of "why is my softmax NaN."

RMSNorm

A normalization layer Llama uses instead of LayerNorm. Cheaper, surprisingly effective.

r   = sqrt(mean(x²) + eps)
out = (x / r) * weight

Compared to LayerNorm: no subtraction of the mean, no bias term. Two fewer arithmetic operations per element, no quality loss on big models.

RoPE (Rotary Position Embedding)

How modern transformers tell the model where each token is in the sequence. Instead of adding a "position vector" to each token (the old way), RoPE rotates pairs of dimensions in the Q and K vectors by an angle that depends on position.

// for each pair of dims (2i, 2i+1), at position pos:
angle = pos * theta^(-2i / head_dim)
c, s  = cos(angle), sin(angle)
new_0 = x[2i]   * c - x[2i+1] * s
new_1 = x[2i]   * s + x[2i+1] * c

Why rotation? Because the dot-product attention between two tokens at positions p and q ends up depending only on p - q, not on the absolute positions. The model gets relative position information for free, with no extra parameters.

SwiGLU activation

The non-linearity inside the MLP block. It's actually two linear projections combined with the SiLU activation:

mlp(x) = down( silu(gate(x)) * up(x) )

silu(z) = z * sigmoid(z) = z / (1 + exp(-z))

SwiGLU performs better than plain ReLU or GeLU at scale. The "gate × up" pattern lets the network learn which features to suppress per-position before the down-projection.

Residual add

Boring but essential: out = a + b, elementwise. Every layer's output is added back to its input. this is what lets gradients flow through a 32-layer (or 96-layer) stack during training, and at inference it's just two reads and a write.

The order they appear in one transformer block

x_in       = x[t]
x_norm     = RMSNorm(x_in)
q          = matmul(W_q, x_norm)
k          = matmul(W_k, x_norm)
v          = matmul(W_v, x_norm)
q, k       = RoPE(q, k, position = t)
attn       = softmax(q · K^T / sqrt(d)) · V
attn_out   = matmul(W_o, attn)
x          = residual_add(x_in, attn_out)

x_norm2    = RMSNorm(x)
gate       = matmul(W_gate, x_norm2)
up         = matmul(W_up,   x_norm2)
mlp_out    = matmul(W_down, silu(gate) * up)
x          = residual_add(x, mlp_out)

That's one layer. Stack 22 of those (TinyLlama) or 32 (Llama-7B) and you have a model. The math really is just this.

what i learned

  • Numerical stability is sneaky. Softmax without the max-subtraction trick produced NaNs on perfectly innocent prompts the first time long contexts hit. It looks correct on small inputs.
  • RoPE has two conventions and the model files don't tell you which. HuggingFace pairs dim i with dim i + head_dim/2: llama.cpp pairs 2i with 2i+1. Same weights, different math, both look "plausible". output stays coherent for 5–10 tokens then drifts. Painful bug.
  • Don't optimize the CPU kernels. You're moving to GPU. Make them correct, easy to read, easy to compare bit-for-bit against PyTorch. That's the entire job.
  • Row-major vs column-major: pick one and write it down. The cuBLAS step later assumes column-major by default. Get the CPU side aligned now and you save a week.
  • SwiGLU is just three matmuls and a sigmoid. The fancy name does it no favors. it's elementwise after the matmuls. The "wow" comes from the gate pattern, not the activation function itself.

What's in the repo
