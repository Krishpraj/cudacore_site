Stage 5 — just stream to stdout

This sounds trivial but is the moment your project starts feeling real. Same generation loop as before, but instead of buffering 200 tokens and printing them all at once, you decode and flush each one as it appears.

for step in 0..max_new_tokens:
    logits    = model.forward(...)
    next_id   = sampler.sample(logits)
    text      = tokenizer.decode_incremental(next_id)
    fwrite(text, stdout); fflush(stdout)
    if next_id == EOS: break

That's the entire feature. No new algorithms. But seeing tokens appear in real time — instead of waiting 30 seconds for a paragraph — changes how the project feels.

¶
analogy

Imagine ordering food at a restaurant. "Streaming" is the waiter bringing each course as it's ready. "Non-streaming" is them holding everything in the back until your entire meal is plated, then dropping it all at once. The food is identical; the experience is night and day.

Incremental decode — the trap

You can't naively tokenizer.decode([single_id]) and concatenate. BPE tokens are not guaranteed to be valid UTF-8 on their own. A Japanese character or emoji might span 2–3 BPE tokens, and decoding one in isolation gives you half a character — which renders as �.

The fix: keep a buffer of recent token IDs. After each new token, decode the whole buffer, emit only the bytes you haven't emitted before, and check if the tail of your output ends in a valid UTF-8 sequence. If not, wait one more token.

Stage 7 — HTTP serving

The transport: Server-Sent Events

OpenAI's /v1/chat/completions with stream: true uses SSE — a one-way streaming protocol over plain HTTP. Each event is one line: data: <json>\n\n, and the stream ends with data: [DONE]\n\n. WebSockets work too but are over-engineered for one-direction streaming.

Matching the OpenAI API is worth the effort. Every client library, terminal tool, and frontend already speaks it. Your server becomes a drop-in replacement.

Continuous batching — the only real optimization that matters at this stage

Naive serving: request comes in, generate to completion, return, then handle the next. GPU sits 90% idle.

Static batching: wait for N requests, run them together. Better, but they all have to finish before any one returns, and short requests get held up by long ones.

Continuous batching (vLLM, TGI's approach): on every decode step, the scheduler picks which sequences to advance. Finished sequences leave the batch immediately and return to their caller. New sequences can join mid-step. The GPU stays full; per-request latency stays low. This is the modern production approach.

Prefill vs decode interleaving

A new request arrives. Its prefill is one big compute-bound matmul over the whole prompt. Existing requests are doing tiny memory-bound single-token decode steps. If you naively run the prefill first, every existing user pauses for hundreds of milliseconds. The trick is "chunked prefill" — split the prefill into smaller chunks that can be interleaved with decode steps in the same batch.

Slot management

With a paged KV cache (Stage 7c), you have a fixed pool of cache pages. The scheduler must:

  • Refuse new requests when the pool is full, or evict idle ones (and rerun their prefill if they come back).
  • Free a request's pages immediately on completion or client disconnect.
  • Handle request timeouts.

Backpressure

What if the client reads slower than your model generates? Options:

  • Drop tokens — rude.
  • Block the model loop until the socket drains — wastes GPU.
  • Bounded buffer with cancellation if it fills — what real systems do.

For a single-user demo, just block. For production, you eventually need a bounded buffer per request plus cancellation when a client disconnects (you'd be surprised how often that happens at scale).

// what I learned

  • Streaming is a UX feature disguised as a backend feature. Same total time, but the perceived latency drops to "time until first token." A 5-second wait feels like 0.5 seconds if the first token shows up at 0.5.
  • Always fflush. stdout is line-buffered by default, so short outputs without newlines look broken — the tokens appear all at once at the end. Pure footgun.
  • SSE > WebSocket for token streaming. Lighter, simpler, works through firewalls, matches what every existing client expects.
  • Continuous batching is what lets one model serve many users economically. Without it, your GPU sits idle most of the time. With it, you can serve 8–16 concurrent generations and aggregate tokens/sec scales nearly linearly.
  • Client disconnects are common. If you don't free their KV cache slot on disconnect, you slowly leak memory and run out of slots at the worst moment.
  • Timeouts kill streams silently. Default server timeouts are often 30s. Long generations need timeouts in minutes, or the connection dies mid-stream with no useful error to the client.
  • CORS bites browser clients. First time you hit your API from a webpage, fetch fails for no visible reason. Add the headers.
