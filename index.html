01 // what is an inference engine?

The thing between a prompt and a reply.

When you type into ChatGPT and watch tokens stream back, an inference engine is what's running. Training built the model. Inference is the act of using it.

¶
analogy for newbies

A trained model is a giant book of numbers. An inference engine is the librarian who can read that book one page at a time — fast enough that you don't notice — and write you back a sentence that makes sense.

An LLM is autoregressive: it predicts the next token given everything so far, then appends that token to the input and predicts the next one. The "engine" is the machinery that does this loop correctly, fast, and for many users at once. Every component on this site exists because some part of that loop needs to happen, or needs to happen faster.

02 // hold this in your head the whole time

The loop, written in plain code.

Every box on this site maps to a line below. Memorize the loop and the rest is just engineering.

# prompt comes in as text prompt = "The cat sat on" # turn text into integers the model can read tokens = tokenize(prompt) → cudacore_tokenizer for step in range(max_new_tokens): # do the giant matrix-multiply pass through the transformer logits = model.forward(tokens, cache) → cudacore_model + cudacore_kernels + cudacore_cuda # remember the new key/value vectors so we don't recompute them next step cache.append(...) → cudacore_kv_cache # turn the model's score vector into a single chosen token next_token = sample(logits[-1]) → cudacore_sampler # show that token to the user immediately emit(next_token) → cudacore_server tokens.append(next_token) if next_token == EOS: break

That's it. The forward pass is where 99% of the time goes. Everything else exists either to feed it, store its state, choose from its output, or share it across many users.

03 // the seven pieces

Each box is its own repo, its own page.

Read in order if you're new — each component teaches a concept the next one builds on. Skip around if you know what you're after.

stage 1 · easiest

Tokenizer

Turns text into integer IDs and back. The model never sees letters — only numbers. This is the translator at the door.
stage 2 · core

Model (CPU forward pass)

The actual transformer: embeddings, attention, MLP, norms, residual connections. Slow but correct. Everything else is an optimization of this.
stage 3 · biggest win

KV cache

Stop recomputing the past on every step. Turns O(n²) generation into O(n). The single most impactful optimization in inference.
stage 2 · foundations

Kernels (CPU math)

The pure-C++ building blocks: matmul, softmax, RMSNorm, RoPE, SwiGLU. Reference implementations the GPU versions will be compared against.
stage 4 · the big one

CUDA kernels

The same math, on the GPU. cuBLAS for matmuls, hand-written kernels for the small ops, fused attention for the win. Where "fast" lives.
stage 2 → 5

Sampler

Given a vector of scores over the whole vocabulary, pick one token. Greedy, temperature, top-k, top-p. Tiny code, huge effect on output style.
stage 5 → 7

Server

The outermost loop: stream tokens to a terminal, then to a browser, then to many browsers at once with continuous batching.
04 // the roadmap

Build it in layers. Don't skip a stage.

Each stage has a definition of done. Numbers don't move on until correctness is proven.

stage 0
Foundations

Read the original attention paper, RoPE, RMSNorm, SwiGLU. Read llama2.c twice. Know what a warp is.

stage 1
Tokenizer

Round-trip text → tokens → text, matching HuggingFace byte-for-byte.

stage 2
CPU forward pass

Load real weights, generate text on CPU in fp32. Identical first 50 greedy tokens vs transformers.generate.

stage 3
KV cache

Same outputs, but per-token latency flat across context length.

stage 4
GPU port

cuBLAS for matmul, custom kernels for the small ops, fused attention. Same outputs, 4× to 100× faster.

stage 5
Streaming

Tokens print as they're generated. UTF-8 multi-byte sequences render correctly.

stage 6
Quantization

fp16, int8, int4. Shrink weights so a 7B model fits on a laptop GPU.

stage 7
Batching & serving

Continuous batching, paged KV cache, HTTP API. Serve many users from one model.
